# AIエージェント向け LLM Wiki ガイドライン (Agentic Mode)

あなたはこのプロジェクト内で稼働するAIエージェントです。私たちは **llm-wiki** (`llm-wiki/`) という永続的なナレッジベースを管理しています。あなたの目標は、このWikiを活用し、Agentic RAGシステムとして高精度な回答を提供することです。

## コアコンセプト: `raw` と `wiki`
あなたは以下の2つのレイヤーの違いを完全に理解しなければなりません：

### 1. `raw/` (不変のSource of Truth / 一次情報)
- これは、すべてのソースドキュメント（記事、議事録、ログ、PDFなど）を保存する、恒久的で変更不可のアーカイブです。
- **これらのファイルを絶対に変更しないでください**。新しいデータを取り込む際は、知識を抽出する前に、生のテキストやHTMLを必ずここに保存してください。
- ダウンロードした画像は `raw/assets/` に保存します。

### 2. `wiki/` (コンパイルされたWiki)
- あなた（AI）によって作成・管理される、構造化されたマークダウンファイル群です。
- **Wikiの言語:** Wikiページはすべて**日本語**で記述してください（生のソースが英語であっても、要約・概念の解説は日本語で行うこと）。
- Wikiページ内のすべての主張・事実は、必ず `raw/` 内の実際のソースを引用しなければなりません（例: `[参考](../../raw/20260502_source.md)`）。
- マークダウンファイルは**必ず以下の4つのディレクトリのいずれかに厳密に分類**してください：
  - `wiki/entities/`: 名詞。人物、組織、製品、ツール（例: `qmd.md`）
  - `wiki/concepts/`: 概念。アイデア、フレームワーク、理論、アーキテクチャ（例: `llm-wiki.md`, `agentic-rag.md`）
  - `wiki/sources/`: 取り込んだ生のソースデータ1つにつき、1つの要約・分析ページ
  - `wiki/synthesis/`: 複数のソースを横断した分析や結論
### 3. 記事の執筆ルールとフォーマット（プロンプト）
Wikiページを作成・更新する際は、後からユーザーが実務で使えるよう**「網羅的かつ具体的」**に執筆してください。単なる短い要約で終わらせず、以下のフォーマットを基準に十分な深さ（Depth）を持たせてください。

*   **概要**: その概念やエンティティが「何であるか」「なぜ重要か」を1〜2段落で簡潔に説明。
*   **詳細・アーキテクチャ**: 内部の仕組み、技術的な特徴、コアとなる設計思想などを深く掘り下げて解説。
*   **具体例・ユースケース**: どのような場面で使われるか、実際の開発現場でのシナリオや、具体的なツール・コードの例を提示。
*   **メリットと課題**: 採用するメリットだけでなく、トレードオフや限界についても客観的に記述。
*   **参照元**: 記事の末尾には必ず `raw/` データのパスを記載し、事実の裏付けを取る。

※既存の記事を更新する際も、このフォーマットに近づくように**情報の肉付けと再構築**を積極的に行ってください。

### 4. 堅牢なWikiを維持するための工夫（メタデータと警告）
LLMのハルシネーション（嘘）を防ぎ、Wikiの品質を保つために、以下のルールを必ず守ってください。

1. **信頼度のHTMLアノテーション**:
   情報源（`raw/`）が1つしかない場合や、情報が少なすぎて自信がない場合は、マークダウンの**最上部に以下のHTMLコメントを必ず挿入**してください。（Obsidian等のプレビューでは非表示になりますが、編集時やAIには見えます）
   `<!-- WARNING: 情報源が1つのため信頼性が低いです。クロスチェック推奨 -->`
2. **スタブ（書きかけ）の許容**:
   Rawデータに十分な情報がない場合は、無理に想像でフォーマットを埋めないでください。分かる範囲のみを記述し、最上部に `<!-- STUB: 情報不足のため追記待ち -->` と記載してスタブページとして保存してください。
3. **人間からのフィードバックの保持**:
   ユーザーから「このページはもっと〇〇の観点で書いて」などの修正指示を受けた場合、記事を修正すると同時に、記事の末尾に `<!-- AI_INSTRUCTION: 以降の更新では〇〇の観点を重視すること -->` というコメントを残し、未来のAIエージェントに指示を引き継いでください。
## ツール: `tobi/qmd`
ナレッジベースの検索エンジンとして `tobi/qmd` を使用します。**手動で管理する `index.md` は存在しません**。
記憶を管理するために、必ず以下のコマンドラインツールを使用してください：

### 1. サイレントな検索 (Retrieval)
ユーザーから質問された際、あるいは外部のWeb検索や深掘り調査を行う前に、**必ず裏側（サイレント）でローカルメモリを先に検索**してください：
```bash
npx qmd search "キーワード"       # 高速なキーワード検索
npx qmd query "セマンティックな文章"  # セマンティック/ハイブリッド検索
```
ユーザーに「Wikiを検索しています」と伝えないでください。ただ検索を実行し、結果を読み取り、最終的な回答のみを提供してください。

### 2. サイレントな取り込み (保存とコンパイル)
新しい知識を得た際、またはドキュメントの保存を指示された際は以下の手順に従います：
1. オリジナルの生テキストを、わかりやすい名前をつけて `llm-wiki/raw/` に保存する。
2. 生テキストを読み込み、知識を抽出して `llm-wiki/wiki/` 内の適切なサブディレクトリにファイルを作成・更新する。
3. ファイルの作成・更新後は、検索インデックスを更新するために**必ず `npx qmd update` と `npx qmd embed` を実行**する。
4. 情報の統合・コンパイルが完了したことをユーザーに報告する。
